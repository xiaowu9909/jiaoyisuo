## 四、前端安全（frontend/src/api/index.ts）

| 问题 | 状态 | 代码位置 / 说明 |
|------|------|-----------------|
| CSRF 存 localStorage | **已修复** | CSRF 仅存模块内变量 `_csrfToken`，`getCsrfToken`/`setCsrfToken` 不写 localStorage（约 L21–29） |
| Promise<any> 类型虚设 | **未改** | 仍为 `Promise<any>`，需逐步补全返回类型 |
| 无 AbortController | **已修复** | `RequestOptions` 含 `signal?: AbortSignal`，`fetch` 传入 `options.signal`（约 L41、59） |
| 错误信息泄露 | **已修复** | `sanitizeErrorMessage` 截断并脱敏后端异常关键词后再 `message.error`（约 L14–18、85–86） |

**验证 3（XSS/CSRF）**：当前 CSRF 不在 localStorage，篡改 localStorage 不会影响服务端校验（服务端以 Session 中 token 为准）。在控制台执行下面语句**不会**绕过 CSRF：

```javascript
localStorage.setItem('vaultpi_csrf_token', 'hacked_token');
// 然后发起 POST：请求头 X-CSRF-TOKEN 仍由前端从内存 _csrfToken 读取，且服务端校验的是 Session 中的 token
```

若要验证「内存存储」：登录后不往 localStorage 写 CSRF，刷新页面后需重新登录或从 /check/login 拿新 token，属预期行为。
