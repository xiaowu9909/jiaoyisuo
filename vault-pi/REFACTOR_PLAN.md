# Vault π 新项目重构方案

> 用新技术栈重做老项目全部功能，老项目仅作业务参考，不直接复用其代码。

## 一、技术栈选型

### 后端（单体优先，后续可按需拆微服务）

| 项目     | 老项目           | 新项目                    |
|----------|------------------|---------------------------|
| 框架     | Spring Boot 1.5/2.7 + 多模块微服务 | **Spring Boot 3.2** 单体 |
| JDK      | 8 / 17 混用      | **JDK 17**                |
| 数据库   | MySQL + JPA      | **MySQL 8** + **Spring Data JPA** |
| 缓存     | Redis            | **Redis**                 |
| 消息队列 | Kafka + Zookeeper | **可选：Kafka** 或先内存/Redis |
| 认证     | Shiro + Session  | **Spring Security + JWT** 或 Session |
| 构建     | Maven 多模块     | **Maven 单模块**          |

### 前端

| 项目     | 老项目        | 新项目               |
|----------|---------------|----------------------|
| 框架     | Vue 2 + iView | **Vue 3**             |
| 构建     | Webpack 3/4   | **Vite 5**            |
| 状态     | Vuex          | **Pinia**             |
| 请求     | vue-resource  | **axios**             |
| UI       | iView         | **Element Plus** 或 Naive UI |
| 语言     | 无            | **TypeScript**（推荐）|

## 三、实施阶段

### Phase 0：新项目骨架（当前）

- 新建仓库/目录：`vault-pi`（与老项目同级或独立）
- 后端：Spring Boot 3.2 单体，包结构 `user / content / market / exchange / admin`，MySQL + Redis，健康检查
- 前端：Vue 3 + Vite，路由 + Pinia + axios，布局 + 首页 + 登录/注册占位 + 交易页占位

### Phase 1：核心可跑（目标：能登录、能看行情、能下单）

- 用户：注册（邮箱）、登录（邮箱/用户名）、登出、当前用户
- 内容：公告分页、轮播广告、帮助分类（可先只读）
- 行情：交易对列表、行情快照、K 线（可先模拟或从老库同步）
- 交易：下单、撤单、当前委托、历史委托（撮合可先简化或对接老 exchange 引擎）
- 前端：首页、登录/注册、交易页（行情+下单+委托）

### Phase 2：资产与个人中心

- 钱包：资产列表、资金流水、充值/提现（可先只做记录，不做链上）
- 个人中心：安全设置、提现地址、邀请页

### Phase 3：管理端

- 管理员登录、菜单权限
- 会员、内容、财务、币种与交易对、系统配置

### Phase 4：活动、红包、CTC、OTC（按需）

- 创新实验室、红包、CTC、OTC 等，业务逻辑照老项目参考，接口与前端重写。

## 五、与老项目的关系

- **不**直接拷贝老项目 Java/Vue 源码，仅参考：
  - 接口 URL 与入参出参（可适当简化）
  - 表结构与业务规则（建表可参考老 SQL，字段可精简）
- 老项目保留在 `CoinExchange_CryptoExchange_Java`，用于对照与数据迁移参考。
- 新项目从零依赖、从零配置，避免历史债。

---

## 六、落地顺序（建议）

1. 在本方案基础上，创建 `vault-pi/backend` 与 `vault-pi/frontend` 骨架。
2. 实现 Phase 1（注册/登录、公告/广告、行情、下单/撤单/委托），保证前后端联调通过。
3. 再按 Phase 2 → 3 → 4 逐步补齐，每阶段可独立交付验证。
