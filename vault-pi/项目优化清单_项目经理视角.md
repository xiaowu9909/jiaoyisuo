# 项目优化清单（项目经理视角 · 用户侧）

从**用户可感知**、**业务可靠性**和**运维可执行**三个维度，列出当前项目仍可优化的项，供排期与迭代参考。已在前序会话中完成的加固（原子性、批量容错、撮合异常隔离、WebSocket 心跳、生产配置等）不再重复。

---

## 一、用户可感知的体验

| 优先级 | 问题 | 用户影响 | 建议方案 |
|--------|------|----------|----------|
| 高 | **成交通知未触达** | 用户下单成交后不知道，只能刷新或反复查委托 | 已有 `OrderTradeEvent` + `@Async` 监听；可接 WebSocket 推送或站内信，前端订阅后 toast/红点 |
| 高 | **错误提示不友好** | 限流、系统忙时只看到「系统繁忙」或技术错误码 | 统一错误码与前端文案映射；限流时明确「操作过于频繁，请 X 秒后重试」 |
| 中 | **长时间无反馈** | 下单/撤单在高峰或 DB 慢时无 loading 或超时提示 | 前端对关键接口设合理超时 + AbortController；超时后提示「请求超时，请查看委托是否已成交」 |
| 中 | **行情断线无感知** | WebSocket 断开后页面仍显示旧价，用户误以为实时 | 前端监听连接状态，断线时提示「行情连接已断开，正在重连」并禁止下单或提示可能延迟 |
| 低 | **多语言不完整** | 错误信息、表单校验多为中文，国际化不全 | 将后端返回的 message 与前端 i18n key 对应，或后端按 Accept-Language 返回文案 |

## 二、可靠性与数据完整性

| 优先级 | 问题 | 业务影响 | 建议方案 |
|--------|------|----------|----------|
| 高 | **停机时未刷批** | 进程被 kill 时，最近 100ms 内的成交只存在于内存批次，未落库 → 资金与订单可能短暂不一致 | 增加优雅关闭：`@PreDestroy` 或 `ApplicationListener<ContextClosedEvent>` 中调用 `MatchBatchPersistence.flush()`，并给 5–10s 缓冲时间再退出 |
| 高 | **批量失败无告警** | `vaultpi.batch.persistence.failures` 已打点，但未接 Grafana/钉钉等告警 | 配置告警规则：该指标 > 0 即告警；并写运行手册「批量持久化失败时先查 DB/网络，再考虑重放或对账」 |
| 中 | **重启后订单簿与 DB 时间差** | 重启瞬间加载的是 DB 中 TRADING 订单，停机前未 flush 的成交会丢失（已通过原子性修复避免「钱动单不动」） | 文档中说明：正常应优雅停机；若异常宕机，可能存在极少量未落库成交，需对账脚本或人工核对 |
| 中 | **无订单/撤单幂等提示** | 用户重复点击可能看到「重复请求」但不知道是否已成功 | 幂等键返回时若命中已有订单，接口明确返回「该请求已处理，订单号 xxx」并在前端 toast |
| 低 | **虚拟盘 Redis 丢失** | Redis 清空后虚拟盘从配置中心价重启，历史 K 线缺口 | 已在 DISASTER_RECOVERY 中说明；可考虑将关键快照同时写 DB 或冷备 |

## 三、运维与可观测性

| 优先级 | 问题 | 运维影响 | 建议方案 |
|--------|------|----------|----------|
| 高 | **健康检查未包含撮合** | `/actuator/health` 有 DB、Redis，但不体现「订单簿是否已加载」 | 自定义 HealthIndicator：检查 `MatchService` 是否已执行 `loadOrderBooks()`，或队列积压是否超过阈值 |
| 高 | **无请求级追踪** | 排查「某笔订单为何未成交」时难以串联日志 | 引入 Sleuth/Micrometer Tracing，或至少在每个请求入口设置 traceId 并贯穿日志与异步任务 |
| 中 | **日志未按环境区分** | 生产若打 DEBUG 会泄露信息且影响性能 | 已在 application-prod 配置 logging.level；建议再收紧 SQL 与第三方库为 WARN，并确保敏感字段脱敏 |
| 中 | **部署与迁移无清单** | 新人或运维不清楚「上线前必须执行 Flyway、必须设 prod profile」 | 在 README 或 scripts/CHECKLIST 中写「上线前检查项」：Flyway、Redis、profile、默认管理员关闭等 |
| 低 | **监控看板未固化** | Prometheus 有指标，但无现成 Grafana 大盘 | 导出一份 JSON 大盘：核心包含 batch.queue.size、match.exceptions、login/order failures、JVM/DB 连接池 |

## 四、产品与业务规则

| 优先级 | 问题 | 产品影响 | 建议方案 |
|--------|------|----------|----------|
| 中 | **单笔订单上下限未配置** | 用户可下 0.00000001 或极大数量，影响撮合与展示 | 在 `ExchangeCoin` 或配置中增加 minAmount/maxAmount、minNotional；下单前校验并返回明确提示 |
| 中 | **自成交规则未对用户可见** | 当前禁止自成交，但前端无说明 | 在「委托说明」或帮助文档中写「同一账户的买单与卖单不会互相成交」；若未来开放，再提供开关说明 |
| 低 | **撤单结果不明确** | 部分成交后撤单，用户可能不清楚「剩余数量已撤」 | 撤单接口返回当前订单状态与剩余数量；前端展示「已撤单，剩余 xxx 已取消」 |
| 低 | **无交易时间/维护窗口** | 维护时若未停机，用户仍可下单 | 可增加「全局维护开关」或「交易对暂停交易」配置，接口层直接拒绝并返回维护公告 |

## 五、安全与合规

| 优先级 | 问题 | 风险 | 建议方案 |
|--------|------|------|----------|
| 中 | **敏感操作无审计日志** | 登录、改密、大额提现等仅依赖 DB 与业务表，不便审计 | 单独写审计表或日志（user_id、action、time、ip、结果）；管理端操作（改趋势、改配置）已有 virtual_trend_audit，可统一规范 |
| 中 | **密码策略未强约束** | 仅前端或部分校验，弱密码仍可注册 | 后端注册/改密时强制：长度、大小写+数字+特殊字符、与用户名/邮箱不重复等，并返回明确错误 |
| 低 | **会话并发未限制** | 同一账号多端登录均有效，若泄露则难以发现 | 可选：同一账号仅允许单端在线，或记录最后登录设备/IP 供用户查看 |

## 六、性能与成本（用户侧体感）

| 优先级 | 问题 | 体感/成本 | 建议方案 |
|--------|------|-----------|----------|
| 中 | **交易对/币种列表每次查库** | 前端频繁拉列表时增加 DB 压力 | 对「当前启用的交易对、币种列表」做短时缓存（如 1 分钟），减少重复查询 |
| 低 | **前端资源未 CDN** | 静态资源走应用服务器，海外或弱网慢 | 生产将 frontend/admin 构建产物部署到 CDN，或 Nginx 配置静态缓存 |
| 低 | **大列表无分页或虚拟滚动** | 委托/成交记录过多时一次拉很多条 | 接口已分页；前端对超长列表做虚拟滚动或「加载更多」，避免首屏卡顿 |

## 七、文档与协作

| 优先级 | 问题 | 影响 | 建议方案 |
|--------|------|------|----------|
| 中 | **API 变更无版本约定** | 前端与后端并行开发时，接口字段变更易踩坑 | 保持 `/api/v1` 稳定；新增或破坏性变更走 v2；在 CONFIGURATION.md 或单独 CHANGELOG 中记录重要配置与接口变更 |
| 低 | **配置项分散** | 部分 vaultpi.* 在 application.yml，部分在 prod | 已建 CONFIGURATION.md；可再补「生产必改项」一节，便于运维一次性核对 |
| 低 | **故障演练缺失** | 未定期模拟 DB 抖、Redis 挂、批量失败 | 在测试环境定期做「停 DB/停 Redis/模拟 flush 失败」演练，验证告警、降级与恢复流程 |

---

## 建议的近期迭代顺序（按用户价值与风险）

1. **优雅关闭 + 批次 flush**（高）：避免停机丢成交，实现成本低。  
2. **成交通知触达**（高）：WebSocket 或轮询「我的成交」，用户体感提升明显。  
3. **批量失败告警 + 运行手册**（高）：出现问题能第一时间发现与处理。  
4. **健康检查含撮合/队列**（高）：便于 K8s/负载均衡做就绪探测。  
5. **错误文案与限流提示**（中）：减少用户困惑与客诉。  
6. **订单上下限与自成交说明**（中）：产品规则清晰、合规更好。  
7. **审计日志与密码策略**（中）：安全与合规补强。  
8. **请求追踪与 Grafana 大盘**（中）：排障效率提升。  

其余项可按资源与排期放入后续版本。如需对某一项做技术方案或直接改代码，可指定条目再细化。
